# 数独游戏 - 作业2 设计文档

---

## 一、对比作业1的改进点
1. **历史记录机制优化**
   从 history/future 双栈结构改为**单历史数组 + 索引指针**，结构更清晰，减少冗余拷贝，Undo/Redo 跳转更稳定。
2. **新增状态能力**
   为 Sudoku 增加 `isComplete()` 方法，支持 UI 判断游戏是否完成。
3. **增加 Svelte 适配层**
   新增 `createGameStore`，作为领域对象与视图之间的桥梁，彻底解决“领域对象与 UI 脱节”问题。
4. **统一状态入口**
   所有操作必须通过 store 方法触发，禁止直接修改内部数据，保证状态可预测、可刷新。

---

## 二、视图层如何消费领域对象
1. **View 直接消费的对象**
   视图不直接使用 Sudoku / Game 实例，而是消费 **gameStore**（自定义 Svelte Store）。
2. **视图可获取的响应式状态**
   - `grid`：当前 9×9 数独盘面
   - `canUndo`：是否可执行撤销操作
   - `canRedo`：是否可执行重做操作
   - `isComplete`：数独盘面是否已填满

---

## 三、Svelte 响应式实现机制
1. **实现方式**
   自定义符合 Svelte 规范的 Store，基于 `subscribe` 订阅模式实现响应式。
2. **更新流程**
   - 用户操作 → 调用 store 方法（guess/undo/redo）
   - 领域对象状态更新 → 执行 `notify()`
   - 推送最新状态给所有订阅者
   - Svelte 通过 `$` 语法感知状态变化 → 自动刷新视图
3. **关键约束**
   禁止直接修改内部数据，必须通过 store 提供的方法操作，否则无法触发界面刷新。

---

## 四、核心交互流程
1. **初始化**
   调用 `createGameStore` 传入初始数独，创建响应式状态源。
2. **用户填数**
   界面输入 → 调用 `gameStore.guess()` → 状态更新 → 界面刷新。
3. **撤销/重做**
   点击按钮 → 调用 `undo/redo` → 切换历史快照 → 界面刷新。
4. **状态判断**
   通过 `isComplete` 自动判断游戏是否完成，控制胜利状态展示。

---

## 五、代码职责边界
1. **领域层（Sudoku/Game）**
   负责数据、规则、历史记录，与 UI 无关。
2. **适配层（createGameStore）**
   连接领域对象与 Svelte 视图，处理响应式订阅与通知。
3. **视图层（Svelte 组件）**
   只负责渲染与事件触发，不处理业务逻辑。
